# 1.2 Arquitectura de Bases de Datos

## 1.2.1 Arquitectura ANSI/SPARC

La **Arquitectura ANSI/SPARC** es un modelo propuesto en 1975 por el comité **ANSI/SPARC** (*American National Standards Institute / Standards Planning and Requirements Committee*) para organizar los sistemas de bases de datos en **tres niveles**.

Su objetivo principal es **separar la forma en que los datos se almacenan físicamente de la forma en que los usuarios los ven**, garantizando independencia de datos.

### Niveles:

- **Externo:**
    - Es la **vista individual** que tiene cada usuario o aplicación.
    - Muestra solo los datos que necesita.
    - Puede haber múltiples vistas externas.
    - Mejora la seguridad y personalización.
    - Conceptual: modelo global
    
    *→ En un instituto, el alumno ve solo sus notas, no las de toda la clase.*
    
- **Conceptual:**
    - Describe la **estructura global de la base de datos**.
    - Define tablas, relaciones, restricciones e integridad.
    - Es independiente del almacenamiento físico.
    - Solo existe **un esquema conceptual**
    
    *→ El sistema del instituto relaciona alumnos, profesores, asignaturas y notas.*
    
- **Interno:**
    - Describe **cómo se almacenan realmente los datos**.
    - Define archivos, índices, métodos de acceso.
    - Se enfoca en rendimiento y almacenamiento
    
    *→ Servidores donde se almacenan las notas en discos duros.*

## 1.2.2 Modelo Cliente-Servidor

El **modelo Cliente-Servidor** es una arquitectura en la que:

- **El cliente** solicita un servicio.
- **El servidor** procesa la solicitud y envía una respuesta.

Es la base del funcionamiento de Internet, aplicaciones web y sistemas de bases de datos modernos.

### Funcionamiento:

1. El **cliente** envía una petición (request).
2. El **servidor** recibe la petición.
3. El servidor procesa la información (por ejemplo, consulta una base de datos).
4. El servidor devuelve una respuesta (response).

### *Ejemplo*

*Cuando buscas **“zapatillas Nike”** en Amazon:*

- *Tu navegador → **Cliente***
- *Servidor de Amazon → **Servidor***
- *Base de datos → donde están almacenados los productos*

## 1.2.3 Tipos de SGDB: SQL vs NoSQL

Los SGBD pueden clasificarse principalmente en **relacionales (SQL)** y **no relacionales (NoSQL)**.

#### A) Bases de datos relacionales (SQL)

Organizan la información en tablas compuestas por filas y columnas. Las tablas se relacionan mediante claves. Se caracterizan por:

- Estructura fija.
- Uso del lenguaje SQL.
- Alta integridad de datos.
- Adecuadas para relaciones complejas.
- Muy utilizadas en sistemas empresariales.

*Tabla de alumnos:*

| ID | Nombre | Nota |
| --- | --- | --- |
| 1 | Ana | 9 |
| 2 | Luis | 7 |

Ventajas de las bases de datos relacionales:

- Gran consistencia.
- Seguridad avanzada.
- Soporte para transacciones.
- Buen manejo de relaciones entre datos.

#### B) Bases de Datos no relacionales (NoSQL)

Son más flexibles y permiten manejar información no estructurada o semiestructurada. Se caracterizan por:

- No dependen necesariamente de tablas.
- Alta escalabilidad.
- Adecuadas para big data.
- Mejor rendimiento en algunos sistemas distribuidos.

Tipos de bases de datos NoSQL:

| **Tipo** | **Ejemplo** | **Uso principal** |
| --- | --- | --- |
| Documentales | MongoDB | Datos JSON |
| Clave-valor | Redis | Caché y rapidez |
| Columnar | Cassandra | Grandes volúmenes |
| Grafos | Neo4j | Redes y relaciones |

#### Diferencias entre SQL y NoSQL

| SQL | NoSQL |
| --- | --- |
| Usa tablas | Usa estructuras flexibles |
| Datos estructurados | Datos estructurados y no estructurados |
| Mayor consistencia | Mayor escalabilidad |
| Relaciones complejas | Procesamiento masivo |
| Lenguaje SQL | Diferentes métodos de consulta |

### C) Modelos híbridos

Muchas empresas utilizan sistemas híbridos, combinando bases de datos SQL y NoSQL.

*Ejemplo:*

- *SQL para operaciones bancarias.*
- *NoSQL para análisis masivo de datos o recomendaciones.*
